1 プロジェクト振り返り

要件・設計面の振り返り 必須
うまくいったこと、難しかったこと、次回改善点を記載

要件面では、「診断」ではなく「来店前の香りづくり準備」として体験全体を設計できた点が良かったです。トップページで期待値を上げ、アンケートで負荷を抑え、グラフで納得感を作り、予約ページで来店意思を固める流れは、各画面の役割分担が明確でした。一方で、質問分岐、グラフ調整、予約導線をすべて連動させる設計は複雑で、仕様整理を先に固めないと実装判断がぶれやすい課題がありました。次回は、画面遷移図とデータの受け渡し仕様を初期段階でより明文化し、バックエンド保存前提の設計まで先に考えて作成に移る方がよかったと感じました。。

デザイン面の振り返り 必須
UI、ワイヤー、見せ方の観点で整理

デザイン面では、静かな高級感と親しみやすさを両立する方向性を各ページで維持できた点が成果でした。アンケートの選択肢画像や香りグラフの見せ方により、抽象的な香りの好みを直感的に扱いやすくできています。改善点としては、予約ページの設計が甘く、初見ユーザーには優先順位が分かりづらくなる可能性があります。重要CTA、入力欄、補足説明の視線誘導をさらに絞り、スマートフォンでの読みやすさも含めて情報整理を強めたいです。

フロントエンド実装面の振り返り 必須
HTML/CSS/JavaScript 実装の学びと反省

フロントエンド実装では、`sessionStorage` を使って複数画面のデータをつなぎ、状態を引き継ぐ構成をシンプルにまとめられた点が良かったです。特にアンケートから予約までをサーバー無しで成立させたことで、導線確認を素早く回せました。一方で、ロジックが各HTML内で分散しており、今後の拡張や保守では見通しが悪くなる懸念があります。全体の共通化分岐条件、最終的な設計までを先に用意しておくべきでした。

バックエンド実装面の振り返り 必須
Java / Spring Boot 前提での反省と改善

バックエンド実装面では、Spring Boot 導入前に必要な API 単位、DB 項目、状態遷移を先に文書化できた点は前進です。一方で、実装着手前の段階では Controller / Service / Mapper の責務分割、バリデーション方式、トランザクション設計をさらに具体化する必要があります。また、実装している途中で不足項目に気づくなど、本来は最初に全てを整えておくべきでした。

全体を通しての学び 必須
今回の制作で得た知見を簡潔に整理

今回の制作を通して、体験設計と画面実装は切り離せず、特に「どこで期待を上げ、どこで迷わせず、どこで納得させるか」を先に決めることが重要だと学びました。また、フロントだけで素早く体験を組み立てる有効性と、その一方でバックエンド連携を後回しにすると設計負債が生まれやすいことも実感しました。

2 技術記事

技術記事タイトル 必須

VSCode・Codexで発生する文字化けに対して、指示テンプレートを追加して改善した話

記事本文 必須

1. はじめに（何をしたかったか）

私は、VSCodeとCodexを使って既存ファイルを効率よく修正したいと考えていました。 特に、HTMLやMarkdown、設定ファイルなど、日本語を含むファイルを安全に編集したいという目的がありました。 しかし実際に作業を進めると、修正そのものよりも、保存後の文字化けや表示崩れの確認に時間がかかる場面がありました。 そこで今回は、文字化けを防ぐための注意点を事前に指示としてまとめ、テンプレート化して運用することで、作業を安定させる取り組みを行いました。

2. 課題（一部置換/完全置換、Shift_JISとUTF-8とBOM付き、PowerShellエンコーディング問題）

最初に難しいと感じたのは、一部置換と完全置換で注意すべき点が異なることでした。 一部置換では、元のファイルの文字コードを維持しなければ、修正箇所以外まで文字化けする可能性があります。 一方で完全置換では、保存時の文字コード指定を誤ると、ファイル全体が正しく読めなくなる危険があります。

また、日本語ファイルにはShift_JIS、UTF-8、UTF-8 BOM付きなど複数の形式があり、見た目だけでは違いが分かりにくい点も課題でした。 さらに、PowerShellでは表示だけが文字化けして見えることがあり、実際にはファイルが壊れていないのに、破損したと誤って判断してしまう可能性もありました。 このように、表示上の問題と実際のファイル破損を区別することが重要でした。

3. 調査（どのタイミングで発生するか）

そこで私は、文字化けがどのタイミングで発生しているのかを切り分けて確認しました。 具体的には、ファイルを開いた時点で崩れているのか、Codexで編集した直後に崩れるのか、保存後に崩れるのか、 あるいはPowerShellで表示した時だけ崩れるのかを順番に確認しました。

その結果、問題はAIが編集したことそのものではなく、読み込み・置換・保存の各段階で文字コードの前提がそろっていないことにあると分かりました。 特に重要だったのは、PowerShell上の表示だけで実ファイル破損と判断しないことです。 まずは表示の問題なのか、それともファイル自体の問題なのかを分けて考える必要があると学びました。

4. 実装（コード付きで解説）

調査結果をもとに、Codexへ渡す指示の先頭に、文字コードに関するテンプレートを追加しました。 内容は次の通りです。

  • 編集前に対象ファイルの文字コードを確認する
  • 一部置換では元ファイルの文字コードを維持する
  • 完全置換では保存時の文字コードを明示する
  • 日本語ファイルではShift_JIS / UTF-8 / UTF-8 BOM付きを区別する
  • PowerShellの表示だけで破損と判断しない
  • 不明な場合は全文再保存より先に確認を優先する
  • 破壊的変更より最小差分編集を優先する

また、PowerShellを使用する場合も、エンコーディングを明示するようにしました。

例：Get-Content .\sample.txt -Encoding UTF8
例：Set-Content .\sample.txt -Value $text -Encoding UTF8

この実装のポイントは、人が毎回注意するのではなく、あらかじめルールとして固定しておくことです。 作業者の記憶や感覚に頼るのではなく、再発防止を仕組みとして組み込むことを意識しました。

5. 結果（どう解決したか）

テンプレートを追加したことで、日本語ファイルの修正時に文字化けが発生する回数は減りました。 仮に崩れて見えた場合でも、すぐに再保存するのではなく、まず表示上の問題か、実体の問題かを確認する流れが定着しました。 また、一部置換と完全置換を分けて考えるようになったことで、不要な全面再保存も減りました。

その結果、作業中の不安が少なくなり、確認手順に沿って落ち着いて対応できるようになりました。 特に、PowerShellの表示文字化けだけで実ファイル破損と決めつけないという意識は、今後の作業でも重要だと感じています。

6. 学び（何を得たか）

今回の経験を通して、文字化けは単に日本語だから起きる問題ではなく、文字コードの前提が一致していないことで発生する問題だと理解できました。 また、不具合が起きた時には、すぐに修正しようとするのではなく、まずどの段階で発生しているのかを切り分けることが大切だと学びました。

さらに、便利なツールを使うほど、前提条件をテンプレートとして明文化しておくことが効果的であると分かりました。 今回の取り組みを通して、開発では単にコードを書く力だけでなく、問題を整理し、再発防止の仕組みを作る力も大切であると実感しました。

4 今後の学習計画

今後学びたい課題・スキル 必須
次の制作に向けた具体的な学習計画

今後は、予約保存 API の実装、DB 設計の実装反映、入力バリデーション、自動テスト整備を優先して学びます。フロント面では、状態管理の整理、共通コンポーネント化、モバイルでの情報設計改善に取り組みます。あわせて、予約率や離脱率といった運用指標をもとに改善判断できるよう、簡易的な計測設計や分析視点も強化したいです。
