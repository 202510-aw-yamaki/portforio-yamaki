1 システム概要

システムの目的 必須

本システムは、来店前のユーザーに対して「診断」ではなく「香りづくりの準備体験」を提供するフレグランスワークショップ導線です。トップページで体験価値を伝え、アンケートで好みの傾向を取得し、5軸グラフで微調整したうえで予約へ接続することで、来店前から香りづくりを始められる状態をつくります。最終的な香りは店頭でスタッフと一緒に仕上げる前提を保ちつつ、スタッフが事前に把握できる情報を整理することも目的としています。

システム構成 必須
  • フロントエンドは HTML / CSS / JavaScript、バックエンドは Spring Boot / Thymeleaf / MyBatis / MySQL で構成
  • 公開画面は `/`、`/questionnaire`、`/questionnaire/step2`、`/graph`、`/reservation`、`/reservation/complete` で構成
  • API は `/api/health`、`/api/reservation-slots`、`/api/reservations`、`/api/questionnaire-results` を提供する
  • STEP1 / STEP2 の一時保持は `sessionStorage` を併用しつつ、graph 以降は API 保存と DB 永続化を行う

4 詳細仕様

トップページ

画面の目的 必須

ワークショップ概要、基本情報、安心材料、予約導線を提示し、初回ユーザーにも再来店ユーザーにも自然に次画面へ進ませる。

STEP1アンケート

画面の目的 必須

5問でベース傾向を取得し、回答は `fragranceStep1Answers` として保持する。

STEP2アンケート

画面の目的 必須

STEP1 の回答傾向に応じて追加質問を出し、`routeCode` と STEP2 回答を保持する。

香りグラフページ

画面の目的 必須

5軸グラフとスライダーで最終調整を行い、`questionnaire-results` 保存 API を呼び出して `resultCode` を発行する。

予約ページ

画面の目的 必須

予約枠 API で日付別予約枠を取得し、来店種別、人数、スタッフ向けメモ、`questionnaireResultCode` を含めて予約登録 API へ送信する。`resultCode` がある場合はアンケート結果 API を優先再取得し、失敗時のみ `sessionStorage` をフォールバックする。

予約完了ページ

画面の目的 必須

`reservationCode` がある場合に予約 API から内容を再取得し、予約日時、来店種別、人数、メモを再表示する。アンケート要約表示は見送り中である。

5 セキュリティ・非機能

セキュリティ観点 必須
  • 公開画面と公開 API は `permitAll`、それ以外は認証必須の Spring Security 構成で動作する
  • 予約 API はバリデーション、404、409、400 を JSON エラー応答で返す
  • CSRF は現状無効化しており、スタッフ画面実装時に認可設計と合わせて見直しが必要
  • 自由記述メモを保存するため、XSS 対策、サニタイズ、ログ出力制御は継続課題である
対応環境 必須

PC / スマートフォン向けレスポンシブ対応を前提とし、Chrome / Edge / Safari / Firefox の最新版系ブラウザを想定する。手動ブラウザ総点検は継続課題として残っている。
