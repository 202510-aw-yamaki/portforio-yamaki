1 ログイン処理のロジック

フローチャート 必須

スタッフ画面は SecurityConfig のフォームログインで保護している。認証ユーザーは DB ではなく、現状は application.yml の値を使った InMemoryUserDetailsManager で構成する。

[スタッフが /staff/login にアクセス]
  -> StaffPageController.showLogin()
  -> 既に認証済みなら /staff/reservations へリダイレクト
  -> 未認証なら login 画面を表示

[ログインフォーム送信]
  -> POST /staff/login
  -> Spring Security が username / password を認証
     -> 失敗: /staff/login?error
     -> 成功: /staff/reservations へ遷移
  -> 以後 /staff/** は ROLE_STAFF 必須
疑似コード 必須
if request.path == "/staff/login" and request.method == GET then
    if authenticatedAsStaff(authentication) then
        return redirect("/staff/reservations")
    end if
    return loginView(withCounts)
end if

if request.path startsWith "/staff" then
    requireRole("STAFF")
end if

formLogin(
    loginPage = "/staff/login",
    loginProcessingUrl = "/staff/login",
    successUrl = "/staff/reservations",
    failureUrl = "/staff/login?error"
)

2 バリデーション処理のロジック

バリデーションの判定フロー 必須
[アンケート結果送信]
  -> QuestionnaireResultForm を @Valid で検証
  -> routeCode 必須
  -> step1Answers / step2Answers / graphAxes 必須
  -> graphAxes の各値は 0..100
     -> エラーあり: ApiExceptionHandler が 400 JSON を返す
     -> エラーなし: QuestionnaireResultService.createResult()

[予約送信]
  -> ReservationForm を @Valid で検証
  -> slotId / slotLabel / visitType / guestCount 必須
  -> staffMemo は 500 文字以下
  -> ReservationService が visitType / slot / status を業務検証
     -> 不正入力: InvalidRequestException
     -> 枠競合: ReservationConflictException
     -> 枠未存在: ResourceNotFoundException
疑似コード 必須
validateBean(form)

if form is ReservationForm then
    slot = reservationSlotMapper.findByDateAndTime(parsedDate, parsedTime)
    if slot == null then
        throw ResourceNotFoundException
    end if

    visitType = visitTypeMapper.findByName(form.visitType)
    if visitType == null then
        throw InvalidRequestException
    end if

    if slot.status not in ["open", "recommended"] then
        throw ReservationConflictException
    end if
end if

on MethodArgumentNotValidException:
    return ErrorResponse(
        code = "VALIDATION_ERROR",
        message = "Request validation failed",
        fieldErrors = bindingResult.fieldErrors
    )

3 検索処理のロジック

予約枠検索のロジック 必須
公開予約枠 API

1. GET /api/reservation-slots?date=yyyy-mm-dd
2. ReservationApiController が LocalDate に変換
3. ReservationSlotService.findAvailableSlots(date)
4. ReservationSlotMapper.findAvailableByDate(slotDate)
5. 取得した ReservationSlot を ReservationSlotResponse に変換
6. slotId = yyyy-mm-dd_HHmm 形式で返す
スタッフ一覧検索のロジック 必須
スタッフ用予約一覧

入力条件:
- visitType
- instructor
- sort
- dir
- page
- size

処理:
1. StaffReservationService.findAllReservations() で全件取得
2. visitType / instructor の一致条件で絞り込み
3. sort と dir で Comparator を切り替え
4. Controller 側でページング計算
5. 追加で today 件数、未連携件数、休眠候補一覧を集計
6. テンプレート staff-reservations に反映

4 権限チェックのロジック

ルート判定のフロー 必須
if path in ["/", "/css/**", "/img/**", "/api/**", "/questionnaire", "/questionnaire_step2", "/fragrance-graph", "/reservation", "/reservation-complete", "/staff/login"] then
    permitAll
else if path startsWith "/staff/" then
    require ROLE_STAFF
else
    permitAll
end if
データアクセス制御のロジック 必須

公開 API はアンケート結果、予約枠、予約確認情報、ヘルスチェックを匿名で扱える。一方で予約一覧、予約詳細、VIP フラグ更新、来店種別・担当者マスタ操作、休眠予約の論理削除は /staff/** に集約し、認証済みスタッフのみ実行できる。

現行実装に ADMIN 専用階層はなく、設計上の権限区分は ROLE_STAFF のみで運用している。

5 状態更新のロジック

予約枠と予約の更新フロー 必須
予約作成時

1. slotId を date + time に分解
2. reservation_slots から対象枠を取得
3. status が open / recommended のみ予約可能
4. updateStatusIfAvailable(id, "reserved") を実行
5. reservations に新規 INSERT
6. API 応答 status は "CONFIRMED" を返す
関連フラグ更新の判定ロジック 必須
